API Rate Limiting

Our API enforces rate limits to ensure fair usage and maintain service reliability for all consumers. This document describes the rate limiting mechanisms, response headers, and best practices for handling rate-limited responses.

Response Headers

Every API response includes rate limit information in the following headers:

X-RateLimit-Limit: 60
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1625097600
Retry-After: 30
  • X-RateLimit-Limit: Maximum requests allowed in the current window
  • X-RateLimit-Remaining: Requests remaining in the current window
  • X-RateLimit-Reset: Unix timestamp when the rate limit resets
  • Retry-After: Seconds to wait before retrying (only on 429 responses)

Handling Rate Limits

Important: Clients that repeatedly ignore rate limits may be temporarily banned. Always implement exponential backoff.

Exponential Backoff Strategy

When receiving a 429 Too Many Requests response, implement the following retry strategy:

  1. Wait for the duration specified in the Retry-After header
  2. If no header present, wait 2^attempt seconds (1s, 2s, 4s, 8s...)
  3. Add random jitter of 0-1 seconds to prevent thundering herd
  4. Maximum retry attempts: 5
  5. Maximum backoff: 60 seconds

Example Implementation

async function fetchWithRetry(url, options, maxRetries = 5) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fetch(url, options);

    if (response.status !== 429) {
      return response;
    }

    const retryAfter = response.headers.get('Retry-After');
    const delay = retryAfter
      ? parseInt(retryAfter) * 1000
      : Math.min(Math.pow(2, attempt) * 1000, 60000);

    await sleep(delay + Math.random() * 1000);
  }

  throw new Error('Max retries exceeded');
}

Best Practices

Tip: Cache responses locally to reduce the number of API calls. Most resources include Cache-Control and ETag headers for conditional requests.
  1. Monitor your usage: Track the X-RateLimit-Remaining header to proactively reduce request volume before hitting limits.
  2. Batch requests: Use bulk endpoints where available instead of making individual requests.
  3. Cache aggressively: Store responses locally and use conditional requests (If-None-Match) to reduce unnecessary API calls.
  4. Use webhooks: Subscribe to webhooks for real-time updates instead of polling.
  5. Implement circuit breakers: If the API is consistently returning errors, back off entirely for a period.
